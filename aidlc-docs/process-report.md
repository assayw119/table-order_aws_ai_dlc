## 3. 사용한 프롬프트 전략

### 전략 ① 점진적 위임 (Progressive Delegation)
초반에는 기술 스택·DB 등 핵심 결정을 직접 선택하다가, 신뢰가 쌓이자 점차 위임 범위를 확대.
- 초반: `[Answer]: C` 처럼 명확한 선택
- 중반: `"추천해줘"`, `"추천에 맡김"` 으로 세부 결정 위임
- 후반: `"지금 계획된 plan을 모두 진행해줘"` 로 전체 실행 위임

### 전략 ② 짧고 명확한 승인 게이트
각 단계 완료 시 `"승인"`, `"ㄱㄱ"`, `"진행해줘"` 등 간결한 신호로 다음 단계 진행.
→ 워크플로우의 승인 게이트와 잘 맞아 불필요한 왕복 최소화.

### 전략 ③ 구조화된 명세 직접 제공
UI 개편 시 화면 구성·디자인 조건을 **번호 매긴 명세**로 직접 작성.
→ "따뜻한 컬러톤", "44x44px", "2열 그리드" 등 구체적 제약으로 의도를 정확히 전달.

### 전략 ④ 작은 단위 반복 개선 (Iterative Refinement)
큰 구현 후 `"− 버튼 추가"`, `"README 꾸며줘"` 같은 작은 후속 요청으로 다듬기.
→ 한 번에 완성하기보다 점진적으로 품질을 높임.

### 전략 ⑤ 환경 규칙 설정
`korean.md` 스티어링 참조 요청으로 이후 모든 응답을 한국어로 고정.
→ 컨텍스트 일관성 확보.

## 4. 잘 작동한 점 / 배운 점

**효과적이었던 패턴**
- 명세를 번호로 구조화하니 의도 전달이 정확했음 (UI 개편)
- "추천해줘" 위임 시에도 AI가 근거를 제시 → 결정 추적 가능
- 단계별 승인으로 잘못된 방향을 조기에 차단

**개선 여지**
- 일부 결정(예: 위임 항목)은 사후에 근거만 확인 → 초기에 선호를 더 밝혔다면 왕복 절감
- 파일 이동 후 화면 미표시 이슈 → 런타임/캐시 문제는 코드와 분리해 진단 필요
